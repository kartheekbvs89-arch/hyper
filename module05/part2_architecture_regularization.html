1. Batch Size: The Implicit Regularizer

After the learning rate, batch size is the deep-learning hyperparameter with the most subtle and most misunderstood effects. Naively, batch size seems like a pure compute knob: bigger batches use more memory and train faster per epoch, smaller batches fit on smaller GPUs. That intuition is correct as far as it goes, but it misses the fact that batch size changes the gradient signal itself. The gradient you compute is an average over a mini-batch, and the variance of that average is inversely proportional to the batch size:

$$ \text{{Var}}\left[ \frac{{1}}{{B}} \sum_{{i=1}}^{{B}} \nabla_\theta \ell(\theta; x_i, y_i) \right] = \frac{{1}}{{B}} \, \text{{Var}}\left[ \nabla_\theta \ell(\theta; x_i, y_i) \right] $$
Eq. 1 — Mini-batch gradient variance scales as 1/B.

This has three practical consequences. First, small batches act as an implicit regularizer. The noise in the gradient prevents the optimizer from settling into sharp minima (which generalize poorly) and instead biases it toward flat minima (which generalize well). This is the famous "generalization gap" — large-batch training reaches lower training loss but worse validation accuracy than small-batch training at the same LR. Keskar et al. (2017) showed this convincingly for image classification.

Second, large batches allow higher learning rates. Because the gradient is less noisy, you can take bigger steps without diverging. The linear scaling rule (section 2) quantifies this. Third, large batches train faster wall-clock but only up to a point. Doubling the batch size from 32 to 64 typically halves the number of steps per epoch and gives you a 2× speedup. Doubling from 4096 to 8192 gives you almost no speedup because you become bandwidth-limited and the per-step compute dominates.

The practical sweet spot for most vision tasks is 32–512. For NLP pretraining it is 256–4096 (with gradient accumulation to fake larger batches). For tabular MLPs, 32–256 is plenty.

Same model, same data, same LR. Small batches (pink) produce a noisy loss curve because each gradient is a noisy estimate. Large batches (blue) produce a smooth curve but may converge to a sharper, worse-generalizing minimum.
📌
The generalization gap is real but contested

Keskar et al. (2017) showed sharp minima generalize worse, and large batches find sharp minima. Later work (e.g., Hoffer et al. 2017, You et al. 2019) argues the gap can be closed with proper LR scaling, warmup, and optimizer choice. The honest summary: small-batch training is more forgiving and tends to generalize well out of the box; large-batch training is faster but requires more careful tuning of LR, warmup, and weight decay to match small-batch generalization.

2. The Linear Scaling Rule

If you increase the batch size by a factor of $k$, you should usually increase the learning rate by a factor of $k$ as well. This is the linear scaling rule introduced by Goyal et al. (2017) for large-batch ImageNet training. The intuition: with $k$ times more samples per batch, the gradient is $k$ times less noisy, so you can take a step $k$ times larger in the same direction without diverging.

$$ \eta_\text{{large}} = k \cdot \eta_\text{{small}}, \quad k = B_\text{{large}} / B_\text{{small}} $$
Eq. 2 — Linear scaling rule (Goyal et al. 2017).

In practice, the rule holds for $k$ up to about 8–16 on ImageNet. Beyond that, you need warmup and possibly sub-linear scaling (e.g., $\sqrt{k}$). The rule also breaks for very small initial LRs — there is a floor below which the model cannot make progress regardless of batch size. Use it as a starting point, not a law.

For Adam-family optimizers, the rule of thumb is more conservative: scale LR by $\sqrt{k}$ rather than $k$, because Adam normalizes the gradient by its second moment and is less sensitive to gradient magnitude.

3. Network Depth vs Width

A neural network's architecture is parameterized by two main structural knobs: depth (the number of layers) and width (the number of units per layer). Both increase the model's representational capacity, but they do so in different ways.

Depth gives you hierarchical features. Each layer composes the previous layer's features, so a 10-layer network can represent functions that would require exponentially more units in a 1-layer network. This is the famous result from Telgarsky (2016): there exist functions that a $k$-layer network of polynomial width can represent, but that a $k-1$ layer network cannot represent without exponential width. For vision and sequence tasks, depth is the primary axis of capacity — ResNet-152 beats ResNet-18 not because it has more parameters per layer, but because it has more layers.

Width gives you raw capacity per layer. A wider layer can memorize more patterns and is less likely to be the bottleneck that prevents the rest of the network from learning. Width also affects optimization dynamics: wider networks have nicer loss landscapes (less ill-conditioned, fewer saddle points), which is why "widening" a ResNet (Wide-ResNet) often matches or beats "deepening" it for the same parameter budget.

For tabular MLPs, the practical default is 3–5 layers with 256–512 units each. Going deeper rarely helps for tabular data and often hurts because tabular features are already "flat" — there is no hierarchy to discover. Going wider helps up to about 1024 units, then plateaus. For vision CNNs, depth is king: 8 layers (VGG-style) is the floor, 50–200 layers (ResNet-style) is the modern default. For Transformers, both matter: BERT-base is 12 layers × 768 width, BERT-large is 24 × 1024.

4. Dropout: Forcing Redundant Representations

Dropout (Srivastava et al., 2014) is a regularization technique that randomly zeros out a fraction $p$ of the activations in a layer during training. At inference time, no units are dropped (or equivalently, all units are scaled by $1-p$ to compensate). The idea is to prevent co-adaptation: if any single unit can be removed without breaking the model, the network is forced to learn redundant representations, which generalize better.

$$ \tilde{{h}}_i = \frac{{m_i \cdot h_i}}{{1-p}}, \quad m_i \sim \text{{Bernoulli}}(1-p) $$
Eq. 3 — Inverted dropout (training time). At inference, m_i = 1 and no scaling is needed.

The dropout rate $p$ is a hyperparameter, and the right value depends on where in the network the dropout is applied. For early layers (close to the input), use $p = 0.1$ to $0.3$ — you want to preserve most of the input signal. For later layers (close to the output), use $p = 0.3$ to $0.5$ — the model has more redundancy here and can afford to lose more units. For embedding layers in NLP, $p = 0.1$ is typical. For attention weights in Transformers, $p = 0.1$ is the BERT default.

Do not use dropout immediately before or after BatchNorm. The interaction is poorly understood and empirically degrades training: dropout injects noise into the activations, which BatchNorm then "fixes" by re-normalizing, but the resulting statistics are biased at test time. The standard pattern is Linear → ReLU → Dropout (dropout after activation), with BatchNorm placed either before or after the activation but never adjacent to dropout.

⚠️
Dropout is mostly a CNN/MLP technique

Modern Transformers use dropout sparingly — typically only on attention weights and the final pre-classifier layer. Modern CNNs (ResNet, EfficientNet) use it almost never, relying on BatchNorm, weight decay, and data augmentation for regularization instead. For tabular MLPs, dropout is still a workhorse and is often the most impactful regularizer.

5. Batch Normalization: Stabilizing Training

Batch Normalization (Ioffe & Szegedy, 2015) normalizes the activations of each layer to have zero mean and unit variance, computed over the current mini-batch. The normalized activations are then scaled and shifted by learned parameters $\gamma$ and $\beta$. This is one of the most impactful architectural innovations in deep learning — it allows you to use higher learning rates, makes the network less sensitive to weight initialization, and acts as a mild regularizer.

$$ \hat{{x}}_i = \frac{{x_i - \mu_\mathcal{{B}}}}{{\sqrt{{\sigma_\mathcal{{B}}^2 + \epsilon}}}}, \quad y_i = \gamma \hat{{x}}_i + \beta $$
Eq. 4 — BatchNorm. μ_B and σ_B² are per-batch statistics; γ and β are learned.

The two BatchNorm hyperparameters are momentum (default 0.1) and eps (default 1e-5). momentum controls how fast the running mean/variance estimates (used at inference) track the batch statistics — higher momentum means slower updates, which is more stable but lags behind distribution shifts. eps is the numerical-stability constant added inside the square root; you rarely need to change it, but for fp16 training it should be increased to 1e-3 or 1e-4 to avoid overflow.

BatchNorm has a subtle interaction with batch size: at very small batch sizes (≤ 8), the per-batch statistics are too noisy and BatchNorm can hurt training. For small-batch training, use GroupNorm (which normalizes over channel groups within a single sample) or LayerNorm (which normalizes over all channels within a sample) instead. These do not depend on batch statistics and behave identically at train and test time.

6. Weight Initialization: He vs Glorot

Weight initialization sets the starting point of optimization. A bad initialization can cause gradients to vanish (too small) or explode (too large) in the first few steps, and the network may never recover. The two standard initialization schemes are:

  • Glorot/Xavier uniform (Glorot & Bengio, 2010): weights drawn from $\mathcal{U}\left[-\sqrt{6/(n_\text{in} + n_\text{out})}, \sqrt{6/(n_\text{in} + n_\text{out})}\right]$. Designed for tanh/sigmoid activations, which have gradients of order 1 near zero. Use this for nn.Linear layers with tanh or sigmoid.
  • He uniform / He normal (He et al., 2015): weights drawn from $\mathcal{U}\left[-\sqrt{6/n_\text{in}}, \sqrt{6/n_\text{in}}\right]$ (or the normal equivalent $\mathcal{N}(0, 2/n_\text{in})$). Designed for ReLU and variants, which zero out half their inputs and so need twice the variance to maintain signal. Use this for nn.Linear or nn.Conv2d layers followed by ReLU.

The wrong initialization is a silent killer. If you put Glorot init on a deep ReLU network, the gradients will quietly vanish, the loss will not decrease for the first 1000 steps, and you will assume the LR is wrong. Always match the init to the activation. In PyTorch, nn.Linear defaults to kaiming_uniform_ (He) with a nonlinearity argument that defaults to 'leaky_relu' — which is close enough to ReLU that most users never notice. But if you swap activations, check your inits.

7. Activation Functions: ReLU and Friends

The activation function is a per-layer hyperparameter that controls the nonlinearity applied between linear layers. The default choice is ReLU ($f(x) = \max(0, x)$) — fast, simple, and almost always good enough. But ReLU has one known pathology: the "dying ReLU" problem, where a unit's weights drift such that the input is always negative, the output is always zero, and the gradient is always zero, so the unit never recovers. About 10–20% of ReLU units in a typical network are dead at the end of training.

The alternatives, in rough order of popularity:

  • LeakyReLU ($f(x) = \max(0.01x, x)$): fixes dying ReLU by giving a small gradient for negative inputs. The slope (0.01 by default) is a hyperparameter but rarely matters; 0.1 is sometimes better. Use this if you observe many dead ReLUs.
  • GELU ($f(x) = x \cdot \Phi(x)$ where $\Phi$ is the standard normal CDF): smooth approximation of ReLU. Used in all modern Transformers (BERT, GPT, ViT). Slightly better than ReLU on NLP and vision-Transformer tasks; not meaningfully different on CNNs.
  • SiLU/Swish ($f(x) = x \cdot \sigma(x)$ where $\sigma$ is the sigmoid): similar to GELU, used in EfficientNet and many modern CNNs. Often a small accuracy bump over ReLU.
  • Mish ($f(x) = x \cdot \tanh(\text{softplus}(x))$): smoother than Swish, occasionally used in YOLO variants. Marginal improvements; not worth the compute cost in most cases.

The practical recommendation: use ReLU for tabular MLPs and old-school CNNs (ResNet, VGG). Use GELU for Transformers. Use SiLU for EfficientNet-family models. Do not over-think it — the activation choice is rarely the difference between a working and a broken model.

🧠 Quick Check

You observe that ~30% of ReLU units in your network have zero activation for every input in the training set. The loss is plateauing well above the expected value. What is the most likely fix?

Show Answer

Dying ReLUs. Switch to LeakyReLU (slope 0.01 or 0.1) or GELU, which both have non-zero gradients for negative inputs. Also check that your learning rate is not too high — large LRs can push ReLU weights into the dead zone irrecoverably. If the problem persists, lower the LR and re-initialize.

8. Weight Decay (L2 Regularization)

Weight decay penalizes large weights by adding an L2 term to the loss: $\mathcal{L}_\text{total} = \mathcal{L} + \lambda \|\theta\|_2^2$. The gradient of this term is $2\lambda\theta$, which is equivalent to multiplying the weights by $(1 - 2\eta\lambda)$ at each step — i.e., "decaying" them toward zero. The decay coefficient $\lambda$ is the hyperparameter.

In modern optimizers, weight decay is often decoupled from the gradient and applied directly to the weights (this is the difference between Adam and AdamW). Decoupled weight decay is theoretically cleaner and empirically better — it does not interact with the optimizer's moment estimates. Use AdamW rather than Adam + L2 whenever possible.

The right value of $\lambda$ depends on the dataset and model. Typical ranges:

  • Tabular MLPs: $\lambda = 10^{-5}$ to $10^{-2}$. Often the most impactful regularizer.
  • CNNs on CIFAR/ImageNet: $\lambda = 10^{-4}$ to $10^{-3}$. Standard.
  • Transformers (AdamW): $\lambda = 0.01$ to $0.1$. Note: AdamW uses a different convention where the weight decay coefficient is multiplied directly, not added to the loss. A value of 0.1 in AdamW is roughly equivalent to $\lambda = 10^{-3}$ in L2-Adam.

Always apply weight decay to weights only, not biases or BatchNorm parameters. These have far fewer parameters and regularizing them usually hurts. In PyTorch, this means grouping parameters into decayed vs not-decayed lists when constructing the optimizer.

9. Putting It All Together: Optuna + PyTorch

The whole point of this module is that batch size, dropout, width, depth, weight decay, optimizer choice, and learning rate are all coupled. Tuning them one at a time is slow and likely to leave you in a local optimum. The right approach is to tune them simultaneously with a Bayesian optimizer like Optuna, using the MedianPruner to kill bad trials early.

snippet.py
# ─── optuna_pytorch_tabular.py ────────────────────────────────────────────────
# Tune batch_size, dropout, width, depth, weight_decay, optimizer, and LR
# simultaneously with Optuna on the MNIST dataset (flat MLP — fast to train).

import optuna
import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader, random_split
from torchvision import datasets, transforms

device = "cuda" if torch.cuda.is_available() else "cpu"
BATCH_TRIAL_EPOCHS = 3

tfm = transforms.Compose([transforms.ToTensor(), transforms.Flatten()])
full = datasets.MNIST("./data", train=True, download=True, transform=tfm)
train_ds, val_ds = random_split(full, [55000, 5000],
                                generator=torch.Generator().manual_seed(42))


# ─── 1. Model factory: architecture depends on trial ─────────────────────────
def build_model(width, depth, dropout, activation):
    act = {"relu": nn.ReLU, "gelu": nn.GELU, "silu": nn.SiLU}[activation]
    layers = [nn.Linear(28 * 28, width), act()]
    for _ in range(depth - 1):
        layers += [nn.Linear(width, width), act()]
        if dropout > 0:
            layers.append(nn.Dropout(dropout))
    layers.append(nn.Linear(width, 10))
    return nn.Sequential(*layers).to(device)


# ─── 2. Optimizer factory: optimizer + LR + weight_decay depend on trial ─────
def build_optimizer(model, name, lr, wd):
    # NOTE: apply weight decay ONLY to weight matrices, not biases.
    decay, no_decay = [], []
    for m in model.modules():
        if isinstance(m, nn.Linear):
            decay.append(m.weight)
            if m.bias is not None:
                no_decay.append(m.bias)
    params = [
        {"params": decay, "weight_decay": wd},
        {"params": no_decay, "weight_decay": 0.0},
    ]
    if name == "adam":
        return optim.Adam(params, lr=lr)
    if name == "adamw":
        return optim.AdamW(params, lr=lr)
    if name == "sgd":
        return optim.SGD(params, lr=lr, momentum=0.9)
    raise ValueError(name)


# ─── 3. Objective: Optuna calls this with a trial, returns val accuracy ───────
def objective(trial):
    # Suggest every hyperparameter in one shot.
    lr          = trial.suggest_float("lr", 1e-5, 1e-1, log=True)
    batch_size  = trial.suggest_categorical("batch_size", [64, 128, 256, 512])
    dropout     = trial.suggest_float("dropout", 0.0, 0.5)
    width       = trial.suggest_int("width", 64, 512, step=64)
    depth       = trial.suggest_int("depth", 2, 5)
    weight_decay = trial.suggest_float("weight_decay", 1e-6, 1e-2, log=True)
    optimizer_name = trial.suggest_categorical("optimizer",
                                                ["adam", "adamw", "sgd"])
    activation  = trial.suggest_categorical("activation", ["relu", "gelu", "silu"])

    # Build everything from the trial's suggestions.
    model = build_model(width, depth, dropout, activation)
    optimizer = build_optimizer(model, optimizer_name, lr, weight_decay)
    criterion = nn.CrossEntropyLoss()

    train_loader = DataLoader(train_ds, batch_size=batch_size, shuffle=True,
                              num_workers=2)
    val_loader = DataLoader(val_ds, batch_size=512, shuffle=False, num_workers=2)

    for epoch in range(BATCH_TRIAL_EPOCHS):
        model.train()
        for x, y in train_loader:
            x, y = x.to(device), y.to(device)
            optimizer.zero_grad()
            loss = criterion(model(x), y)
            loss.backward()
            optimizer.step()

        # Validate and report to Optuna (for pruning).
        model.eval()
        correct, total = 0, 0
        with torch.no_grad():
            for x, y in val_loader:
                x, y = x.to(device), y.to(device)
                correct += (model(x).argmax(1) == y).sum().item()
                total += y.size(0)
        acc = correct / total
        trial.report(acc, epoch)
        if trial.should_prune():
            raise optuna.TrialPruned()

    return acc


# ─── 4. Run the study with MedianPruner ──────────────────────────────────────
study = optuna.create_study(direction="maximize",
                            pruner=optuna.pruners.MedianPruner(
                                n_startup_trials=5, n_warmup_steps=1))
study.optimize(objective, n_trials=50, timeout=3600)

print(f"Best val acc: {study.best_value:.4f}")
print(f"Best params: {study.best_params}")

This study runs 50 trials of 3 epochs each — about an hour on a single GPU. The MedianPruner kills the bottom 50% of trials after epoch 1, saving roughly half the compute. The resulting study.best_params dict gives you a coordinated set of hyperparameters that are jointly optimal, which is qualitatively better than what you would get by tuning each knob separately. The next step (covered in Module 09 — Optuna Deep Dive) is to plot param importances and see which knobs actually mattered — usually LR and weight_decay dominate, and dropout barely matters at all on tabular MNIST.

💡
Why weight decay on biases is a bad idea

Biases have one parameter per output unit, vs. n_in × n_out weights per layer. Regularizing biases with the same coefficient as weights is over-regularizing a tiny number of parameters, and it usually biases the model's outputs toward zero in a way that hurts accuracy. Always split your parameter groups: weights get weight decay, biases and BatchNorm .weight/.bias do not.

🎯 Key Takeaways

  • Batch size is an implicit regularizer: small batches inject gradient noise that biases the optimizer toward flat, generalizable minima. Practical range: 32–512 for vision, 256–4096 for NLP.
  • Use the linear scaling rule: when you multiply batch size by k, multiply LR by k (SGD) or √k (Adam). Add warmup for large k.
  • Depth gives hierarchical features (essential for vision/NLP); width gives raw capacity (helps tabular MLPs). Tabular sweet spot: 3–5 layers × 256–512 units.
  • Dropout forces redundant representations. p=0.1–0.3 early layers, 0.3–0.5 later layers. Do NOT use next to BatchNorm.
  • BatchNorm stabilizes training and allows higher LR. Hyperparams: momentum=0.1, eps=1e-5. Use GroupNorm/LayerNorm for batch_size ≤ 8.
  • Match init to activation: he_uniform for ReLU, glorot_uniform for tanh/sigmoid. Wrong init = vanishing gradients.
  • Activations: ReLU (default), GELU (Transformers), SiLU (EfficientNet), LeakyReLU (fixes dying ReLU).
  • Weight decay on weights only — never biases or BatchNorm. Range: 1e-5 to 1e-2. Prefer AdamW over Adam+L2.
  • Tune them all simultaneously with Optuna + MedianPruner. LR and weight_decay usually dominate; dropout barely matters on tabular.
