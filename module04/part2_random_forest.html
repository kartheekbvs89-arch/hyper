1. From One Tree to a Forest

A random forest is a bagging ensemble of decision trees. Where a single decision tree is grown once on the full training set and predicts via a single recursive partition, a random forest grows hundreds of trees, each on a bootstrap resample of the training set, and averages their predictions. The averaging is what gives the forest its power: it does not reduce bias — the average tree in a random forest has the same bias as a single decision tree — but it dramatically reduces variance, because the trees are only weakly correlated with each other thanks to the bootstrap resampling and the per-split feature subsampling.

This means the hyperparameters of a random forest split cleanly into two categories. First, there are the ensemble-level hyperparameters: n_estimators, max_features, bootstrap, max_samples, oob_score. These control how the ensemble is constructed from its base learners. Second, there are the per-tree hyperparameters: max_depth, min_samples_split, min_samples_leaf, max_leaf_nodes. These are passed straight through to each DecisionTreeClassifier inside the forest, and their semantics are identical to what we covered in Part 1.

If you have not read the Decision Tree article yet, go back and read at least the sections on max_depth and min_samples_leaf before continuing. Everything there applies here too — the only difference is that you are tuning the bias of a single tree, while the ensemble mechanism takes care of the variance. The result is that you can usually afford slightly deeper trees in a forest than you would as a single model, because the variance penalty is muted by averaging.

2. n_estimators: More Trees, Lower Variance

n_estimators is the number of trees in the forest. Its effect on the bias-variance tradeoff is unusual: adding more trees reduces variance without changing bias. Each tree has the same expected prediction (averaged over bootstrap samples), so averaging more of them just shrinks the variance of the average. The variance reduction scales roughly as $1/n$ if the trees were independent, but in practice the trees are correlated (because they share training data and consider overlapping feature subsets), so the reduction is more like $\rho/n + (1-\rho)/n$ where $\rho$ is the average pairwise correlation between trees. The key point: adding trees never hurts, but the marginal benefit shrinks rapidly.

Empirically, the OOB error of a random forest drops fast for the first 50–100 trees, then enters a long plateau where each additional tree produces a negligible improvement. By 300–500 trees, the OOB error is essentially converged, and adding more trees is just wasted compute. The exception is when max_features is set very low, which decorrelates the trees and lets you benefit from more of them — but even then, 1000 trees is rarely worth the cost over 500.

Figure 1 — OOB error vs n_estimators on breast_cancer with max_features='sqrt'. The error drops rapidly for the first ~50 trees, then enters a long plateau. 300–500 trees is the practical sweet spot; beyond that, marginal gains are smaller than the run-to-run noise in the OOB estimate itself.

The standard recommendation is to start with n_estimators=100 for development and quick experiments, then raise it to 300–500 for your final model. Going beyond 500 is rarely worth the additional compute unless you have a specific reason (very low max_features, very large dataset, requirement for very stable feature importances). The plot below shows the typical OOB convergence curve and lets you read off the inflection point.

oob_convergence.py
# ─── oob_convergence.py ───────────────────────────────────────────────────────
# Plot OOB error as a function of n_estimators to visualize the convergence
# behavior and pick a sensible tree count for production. Also demonstrates
# the effect of max_features on convergence speed.

import numpy as np
import matplotlib.pyplot as plt
from sklearn.datasets import load_breast_cancer
from sklearn.ensemble import RandomForestClassifier

X, y = load_breast_cancer(return_X_y=True)

# Tree counts to evaluate
tree_counts = [10, 25, 50, 75, 100, 150, 200, 300, 400, 500, 750, 1000]

plt.figure(figsize=(9, 5.5), constrained_layout=True)

for max_feat in ["sqrt", 0.3, None]:
    oob_errors = []
    for n in tree_counts:
        rf = RandomForestClassifier(
            n_estimators=n,
            max_features=max_feat,
            oob_score=True,
            n_jobs=-1,
            random_state=42,
        )
        rf.fit(X, y)
        oob_errors.append(1.0 - rf.oob_score_)
    plt.plot(tree_counts, oob_errors, "o-", label=f"max_features={max_feat!r}")

plt.xlabel("n_estimators", fontsize=12)
plt.ylabel("OOB error", fontsize=12)
plt.title("OOB error vs n_estimators (breast_cancer)", fontsize=13)
plt.legend()
plt.grid(alpha=0.2)
plt.xscale("log")
plt.savefig("oob_convergence.png", dpi=120)
plt.show()

# Report the marginal improvement of going from 300 to 1000 trees
rf300 = RandomForestClassifier(n_estimators=300, max_features="sqrt",
                                oob_score=True, n_jobs=-1, random_state=42).fit(X, y)
rf1000 = RandomForestClassifier(n_estimators=1000, max_features="sqrt",
                                 oob_score=True, n_jobs=-1, random_state=42).fit(X, y)
print(f"OOB error @ 300 trees:  {1 - rf300.oob_score_:.4f}")
print(f"OOB error @ 1000 trees: {1 - rf1000.oob_score_:.4f}")
print(f"Marginal gain (700 extra trees): {(1 - rf1000.oob_score_) - (1 - rf300.oob_score_):+.4f}")

When you run this, you will see the OOB error drop by a few percent as you go from 10 to 100 trees, then flatten out. The marginal gain of going from 300 to 1000 trees is typically less than 0.005 — well within the noise of the OOB estimate itself. This is why the practical advice is to stop at 300–500: you are spending 3x the compute for less than half a percentage point of accuracy.

💡
Set n_estimators last, set it generously

During development, use n_estimators=100 so your search loop is fast. Once you have fixed the other hyperparameters, retrain with n_estimators=500 for your final model. Because n_estimators monotonically reduces variance without affecting bias, you can always add more trees at the end without re-tuning anything else.

3. max_features: The Key Random Forest Hyperparameter

If n_estimators is the most-tuned random forest hyperparameter, max_features is the most important. It controls how many features each tree considers at every split, and it is what makes random forests different from plain bagging. Without max_features, every tree in the ensemble would consider the same features at every split, pick the same splits, and produce nearly identical trees — averaging them would do almost nothing to reduce variance.

By limiting each split to a random subset of features, max_features forces the trees to be different. Even if one feature is overwhelmingly the most informative, only a fraction of trees will be allowed to split on it at the root; the rest will have to find alternative splits on other features. This decorrelates the trees, which is what gives the averaging its variance-reduction power.

The scikit-learn defaults are sensible starting points: max_features='sqrt' for classification (so each split considers $\sqrt{d}$ features, where $d$ is the total feature count) and max_features=d/3 for regression. Empirical work has shown these to be near-optimal across many datasets, but the exact best value can vary — usually you want to search over ['sqrt', 'log2', 0.3, 0.5, 0.7] and pick the winner. Lower values decorrelate the trees more (which lets you benefit from more estimators) at the cost of higher per-tree bias; higher values let each tree fit the data better but increase correlation.

🔥
max_features and n_estimators interact

Lower max_features decorrelates trees, which means the variance reduction from averaging is larger — but only if you have enough trees to average over. With max_features='sqrt' and 50 trees, you may underperform max_features=None with 50 trees because the per-tree bias increase outweighs the decorrelation benefit. With 500 trees, max_features='sqrt' almost always wins. Always co-tune these two.

4. bootstrap, max_samples, and oob_score

Three more ensemble-level knobs control how the bootstrap resampling works. bootstrap=True (the default) means each tree is trained on a random sample of the training data drawn with replacement, of size equal to the training set. bootstrap=False means each tree sees the full training set — this is rarely useful because it removes most of the variance-reduction benefit of bagging, but it can be appropriate when you have a very large dataset and want each tree to see a different deterministic subsample.

max_samples controls the size of the bootstrap sample. The default None means each sample is the same size as the training set (which, because sampling is with replacement, contains about 63% unique samples and 37% duplicates). You can set it to a float like 0.5 to draw half-size samples, which speeds up training and further decorrelates the trees at the cost of higher per-tree bias. This is a less-commonly-tuned hyperparameter but worth knowing about.

oob_score=True is one of the most underused features of random forests. Because each tree is trained on a bootstrap sample, about 37% of training samples are out-of-bag for that tree — they were not selected in the resample. We can use these OOB samples as a free validation set: for each training sample, predict it using only the trees that did not see it during training, and average those predictions. The resulting OOB score is an unbiased estimate of generalization error, computed without any explicit held-out set or cross-validation loop.

oob_demo.py
# ─── oob_demo.py ──────────────────────────────────────────────────────────────
# Demonstrate that OOB score is a free, unbiased validation estimate.
# Compare OOB score against 5-fold CV and held-out test set on breast_cancer.

import numpy as np
from sklearn.datasets import load_breast_cancer
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import train_test_split, cross_val_score, StratifiedKFold

X, y = load_breast_cancer(return_X_y=True)
X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.2,
                                           stratify=y, random_state=42)

rf = RandomForestClassifier(
    n_estimators=300, max_features="sqrt",
    oob_score=True, n_jobs=-1, random_state=42,
)
rf.fit(X_tr, y_tr)

cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
cv_scores = cross_val_score(rf, X_tr, y_tr, cv=cv, scoring="accuracy", n_jobs=-1)

print(f"OOB accuracy:   {rf.oob_score_:.4f}   (free, computed during fit)")
print(f"5-fold CV acc:  {cv_scores.mean():.4f} ± {cv_scores.std():.4f}  (5 extra fits)")
print(f"Test accuracy:  {rf.score(X_te, y_te):.4f}  (held-out set)")

The three numbers should be within a percent or two of each other. The OOB score is essentially a free 5-fold CV estimate — it costs nothing during training and is computed automatically. Use it during hyperparameter search to get a reliable validation estimate without paying for cross-validation. The only caveat is that OOB is specific to bagging ensembles; for gradient boosting or other models you still need CV.

5. Per-Tree Hyperparameters and n_jobs

Every per-tree hyperparameter from Part 1 — max_depth, min_samples_split, min_samples_leaf, max_leaf_nodes, min_impurity_decrease, criterion — is passed through to each tree inside the forest. The semantics are identical to the single-tree case; the only difference is that you are now tuning the bias of the per-tree learners, while the ensemble mechanism handles the variance. In practice, this means you can usually afford slightly more flexible trees in a forest than you would as a single model.

A common pattern is to leave max_depth=None (let trees grow to their natural depth) and rely on min_samples_leaf to control complexity. This works well because the variance penalty of deep trees is largely absorbed by the ensemble averaging. Another pattern is to set max_depth explicitly to keep individual trees small and fast, which is useful when you need to fit thousands of trees on a large dataset.

n_jobs is not really a model hyperparameter — it controls how many CPU cores to use for fitting and predicting. Set n_jobs=-1 to use all available cores. Because the trees in a forest are independent, training parallelizes essentially perfectly: a 300-tree forest on 8 cores takes roughly the same wall-clock time as a 38-tree forest on 1 core. There is no reason not to use n_jobs=-1 in essentially every situation.

6. Bias-Variance: What Bagging Does and Does Not Do

It is worth being precise about what bagging — the technique underlying random forests — actually does to bias and variance. The folklore is that "random forests reduce variance without affecting bias." This is correct, with one important qualification: the bias being referred to is the bias of the base learner (a single decision tree), and that bias is the same as the bias of a single decision tree trained on the full training set.

The math: let $\hat{f}_1, \ldots, \hat{f}_B$ be the predictions of $B$ trees, each trained on a bootstrap sample. The forest prediction is $\hat{f}_{\text{RF}} = \frac{1}{B} \sum_b \hat{f}_b$. The bias of this average is $\mathbb{E}[\hat{f}_{\text{RF}}] - f = \mathbb{E}[\hat{f}_1] - f$ (the bias of a single tree), because expectation is linear. The variance is $\text{Var}(\hat{f}_{\text{RF}}) = \rho \sigma^2 + (1-\rho) \sigma^2 / B$, where $\sigma^2$ is the variance of a single tree and $\rho$ is the average pairwise correlation. As $B$ grows, the second term vanishes, but the first term (proportional to correlation) does not. This is why max_features matters so much: it reduces $\rho$, which is the only term that survives as $B \to \infty$.

$$ \text{Var}(\hat{f}_{\text{RF}}) = \rho \, \sigma^2 + \frac{1 - \rho}{B} \, \sigma^2 $$
Eq. 1 — Variance of a bagged ensemble. The first term is the irreducible correlation floor; the second term vanishes as B → ∞.

The practical implication: bagging cannot fix a high-bias base learner. If your single decision trees are underfitting (because max_depth is too small or min_samples_leaf is too large), adding more of them to a forest will not help — you will just average a bunch of underfit models. To fix high bias, you have to make the per-tree learners more flexible (raise max_depth, lower min_samples_leaf). Once the per-tree bias is reasonable, the ensemble takes care of the variance.

🧠 Quick Check

Your random forest has n_estimators=500, max_features='sqrt', max_depth=3, and gets 75% test accuracy on a complex dataset. Adding 500 more trees brings test accuracy to 75.1%. What is the most likely problem, and what is the fix?

Show Answer

The model is high-bias: max_depth=3 is too restrictive for the underlying structure, so each tree in the forest is underfitting. Adding more trees does not fix bias (Eq. 1 shows that bias is determined by the per-tree bias, not the ensemble size). Fix: raise max_depth (try 8, 12, None), lower min_samples_leaf, and/or raise max_features. Once per-tree accuracy is reasonable, the ensemble can do its job.

7. feature_importances_ vs Permutation Importance

Random forests expose a built-in feature_importances_ attribute that ranks features by their total impurity decrease across all trees. This is convenient and fast, but it has well-known biases: it inflates the importance of high-cardinality features (because they offer more split points), it underestimates the importance of features that are correlated with other features (because the importance gets split between them), and it is computed on training data, so it can overstate the importance of features that the model is overfitting to.

Permutation importance, by contrast, measures how much the validation score drops when a feature's values are randomly shuffled. It is model-agnostic, works on any scoring metric, and reflects how much the model actually relies on each feature at prediction time. The cost is one extra prediction pass per feature (plus the original prediction), so it is slower than the built-in impurity-based measure but still practical for moderate feature counts.

feature_importance_comparison.py
# ─── feature_importance_comparison.py ─────────────────────────────────────────
# Compare impurity-based feature_importances_ with permutation importance
# on breast_cancer. The two rankings often disagree, and permutation
# importance is the more trustworthy measure for understanding model behavior.

import numpy as np
import matplotlib.pyplot as plt
import pandas as pd
from sklearn.datasets import load_breast_cancer
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import train_test_split
from sklearn.inspection import permutation_importance

cancer = load_breast_cancer()
X, y = cancer.data, cancer.target
feature_names = cancer.feature_names

X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.2,
                                           stratify=y, random_state=42)

rf = RandomForestClassifier(n_estimators=300, max_features="sqrt",
                             n_jobs=-1, random_state=42)
rf.fit(X_tr, y_tr)

# ─── 1. Impurity-based (built-in, fast, biased toward high-cardinality features) ─
imp_imp = pd.Series(rf.feature_importances_, index=feature_names,
                    name="impurity").sort_values(ascending=False)

# ─── 2. Permutation importance on the test set ─────────────────────────────────
perm = permutation_importance(rf, X_te, y_te, n_repeats=10,
                              random_state=42, n_jobs=-1, scoring="accuracy")
perm_imp = pd.Series(perm.importances_mean, index=feature_names,
                     name="permutation").sort_values(ascending=False)

# ─── 3. Compare side by side ───────────────────────────────────────────────────
comparison = pd.DataFrame({"impurity": imp_imp, "permutation": perm_imp}).fillna(0)
comparison = comparison.sort_values("permutation", ascending=False).head(10)
print(comparison.round(4))

# ─── 4. Plot ───────────────────────────────────────────────────────────────────
fig, ax = plt.subplots(figsize=(9, 6), constrained_layout=True)
y_pos = np.arange(len(comparison))
width = 0.4
ax.barh(y_pos - width/2, comparison["impurity"], width,
        color="#38bdf8", label="Impurity (training)")
ax.barh(y_pos + width/2, comparison["permutation"], width,
        color="#f472b6", label="Permutation (test)")
ax.set_yticks(y_pos)
ax.set_yticklabels(comparison.index, fontsize=9)
ax.invert_yaxis()
ax.set_xlabel("Importance")
ax.set_title("Top-10 features: impurity vs permutation importance")
ax.legend()
plt.savefig("feature_importance_comparison.png", dpi=120)
plt.show()

The two rankings will overlap on the top few features (usually "worst radius" and "worst concave points" dominate on breast_cancer under both measures) but diverge on the long tail. Features that the impurity measure rates highly but that permutation importance rates lowly are usually ones the model is overfitting to in training but cannot actually use at test time. Features that permutation importance rates highly but that impurity importance misses are usually ones whose importance was diluted across correlated features in the impurity calculation.

8. Full Tuning Example with RandomizedSearchCV

Let us put it all together with a complete tuning pipeline on breast_cancer. We will use RandomizedSearchCV because the random forest hyperparameter space has both discrete (max_features) and continuous (min_samples_split as a fraction) dimensions, and random search handles mixed spaces more naturally than grid search.

rf_tuning.py
# ─── rf_tuning.py ─────────────────────────────────────────────────────────────
# Full RandomizedSearchCV pipeline for a RandomForestClassifier on
# breast_cancer. Uses OOB-style validation via StratifiedKFold, sweeps
# all the key hyperparameters, and reports the best configuration.

import numpy as np
from scipy.stats import randint
from sklearn.datasets import load_breast_cancer
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import (train_test_split, RandomizedSearchCV,
                                       StratifiedKFold)

X, y = load_breast_cancer(return_X_y=True)
X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.2,
                                           stratify=y, random_state=42)

rf = RandomForestClassifier(n_jobs=-1, random_state=42, class_weight="balanced")

# Search space — modest number of dimensions, focused on the high-leverage knobs
param_dist = {
    "n_estimators":      [100, 200, 300, 500],
    "max_features":      ["sqrt", "log2", 0.3, 0.5, 0.7, None],
    "max_depth":         [4, 6, 8, 12, 16, None],
    "min_samples_split": randint(2, 30),
    "min_samples_leaf":  randint(1, 15),
    "bootstrap":         [True, False],
}

cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
search = RandomizedSearchCV(
    estimator=rf,
    param_distributions=param_dist,
    n_iter=60,
    cv=cv,
    scoring="roc_auc",
    n_jobs=-1,
    random_state=42,
    verbose=1,
)
search.fit(X_tr, y_tr)

print(f"\nBest params: {search.best_params_}")
print(f"Best CV ROC-AUC: {search.best_score_:.4f}")
print(f"Test ROC-AUC:    {search.score(X_te, y_te):.4f}")

Typical best params on breast_cancer land around n_estimators=300, max_features='sqrt' or 0.3, max_depth=None (let trees grow naturally), min_samples_leaf=1–4. Test ROC-AUC is usually around 0.99 — breast_cancer is a relatively easy dataset for tree ensembles. The same pipeline works for harder datasets; just expect the optimal hyperparameters to shift toward deeper trees and smaller min_samples_leaf when the data has more complex structure.

📌
The bias-variance tradeoff is a property of the base learner, not the ensemble

When tuning a random forest, focus your search budget on the per-tree hyperparameters (max_depth, min_samples_leaf) first, because they control bias. Once the per-tree bias is reasonable, set n_estimators=500 and max_features='sqrt' as a near-optimal default. Spend remaining budget on a small sweep over max_features and min_samples_leaf; do not waste it on n_estimators (which is essentially monotone-improving) or on bootstrap/max_samples (which rarely move the needle).

Key Takeaways — Random Forest Hyperparameters

  • n_estimators reduces variance without affecting bias: more is always at least as good, with diminishing returns past 300–500 trees.
  • max_features is the key ensemble hyperparameter: 'sqrt' for classification, 1/3 for regression, are sensible defaults.
  • oob_score=True is free validation: an unbiased estimate of generalization error computed during training, no CV needed.
  • Bagging does not fix high bias: if per-tree learners underfit, the forest will underfit too — make trees more flexible first.
  • Permutation importance > impurity importance for understanding what the model actually relies on at prediction time.
