1. Why Tracking Is the Most Underrated Skill

Imagine you ran 500 hyperparameter trials overnight. The next morning you open your notebook and see a best-validation accuracy of 0.982. Which trial produced it? What hyperparameters did it use? Which random seed? Which version of the training data? Which version of the model code? Can you reproduce trial 342 if the production team asks you to? If you cannot answer all of these questions in under five minutes, you have a tracking problem — and almost every team has a tracking problem until they invest in tooling.

The naïve approach — print statements and a folder of .pkl files — works for 5 trials. It collapses at 50. By trial 200 you will have lost track of which script produced which model, and by trial 500 you will have lost track of which models are even worth considering. The professional approach is to use a tracking system that records, for every trial, the full set of inputs (hyperparameters, code commit, data version, environment) and outputs (metrics, artifacts, logs) in a queryable database with a UI for browsing.

This article covers the three tracking systems you will encounter in production ML: MLflow (the open-source standard), Weights & Biases (W&B) (the commercial SaaS most teams end up on), and Ray Tune (the distributed-tuning framework that does its own tracking when you scale to many machines). We will also cover reproducibility (which is harder than it sounds) and multi-objective optimization (which is the production-grade answer to "I also care about inference latency, not just accuracy").

📌
Tracking is separate from tuning

Tracking systems (MLflow, W&B) record what happened during a run; tuning systems (Optuna, Ray Tune) decide what to try next. They are complementary: you tune with Optuna, and you log every trial to MLflow so you can later answer "what did trial 342 do?" The Optuna↔MLflow integration is a one-line callback, covered in Section 3.

2. MLflow Tracking: The Open-Source Standard

MLflow is a Python library + a tracking server + a UI. The library gives you four functions — start_run, log_params, log_metrics, log_artifact — that record everything about a single training run. The server stores these records in a backend database (SQLite for local, Postgres for production) and a blob store (local filesystem, S3, GCS) for artifacts. The UI lets you browse, filter, and compare runs.

The core abstraction is the run: one execution of a training script, identified by a unique run_id. Inside a run you log parameters (hyperparameters), metrics (numbers that change during training, like loss per epoch), tags (metadata like the git commit), and artifacts (files like the trained model, plots, or the CV fold indices). Every run belongs to an experiment, and every experiment belongs to a tracking URI (which is just the server address).

snippet.py
# ─── mlflow_basic.py ───────────────────────────────────────────────────────────
# Minimum viable MLflow tracking: log params, metrics, and a model artifact.

import mlflow
import mlflow.sklearn
import numpy as np
from sklearn.datasets import load_breast_cancer
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import StratifiedKFold, cross_val_score

# Point MLflow at a local tracking directory (or a tracking server URI)
mlflow.set_tracking_uri("file:./mlruns")
mlflow.set_experiment("xgb_breast_cancer_baseline")

X, y = load_breast_cancer(return_X_y=True)
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

# One run = one hyperparameter configuration
with mlflow.start_run(run_name="gbdt_lr0.1_depth5") as run:
    params = {
        "learning_rate": 0.1,
        "max_depth": 5,
        "n_estimators": 200,
        "random_state": 42,
    }
    model = GradientBoostingClassifier(**params)

    # Log hyperparameters (must be primitives: str, int, float, bool)
    mlflow.log_params(params)

    # Log a tag for the code version
    import subprocess
    git_commit = subprocess.check_output(["git", "rev-parse", "HEAD"]).strip().decode()
    mlflow.set_tag("git_commit", git_commit)
    mlflow.set_tag("model_family", "GradientBoostingClassifier")

    # Train and evaluate
    scores = cross_val_score(model, X, y, cv=cv, scoring="accuracy", n_jobs=-1)
    mlflow.log_metric("cv_accuracy_mean", float(scores.mean()))
    mlflow.log_metric("cv_accuracy_std", float(scores.std()))

    # Fit on all data and log the model as an artifact
    model.fit(X, y)
    mlflow.sklearn.log_model(model, artifact_path="model")

    print(f"Run ID: {run.info.run_id}")
    print(f"CV accuracy: {scores.mean():.4f} ± {scores.std():.4f}")

Open the UI with mlflow ui and navigate to http://localhost:5000. You will see your experiment, your run, and clickable columns for every logged metric and parameter. The model artifact shows up under "Artifacts" and can be downloaded as a pickle file. This is the foundation — every other MLflow feature builds on top of start_run / log_params / log_metrics / log_artifact.

3. MLflow + Optuna: One Callback to Log Every Trial

The Optuna↔MLflow integration is a single callback that automatically creates one MLflow run per Optuna trial, logging the trial's hyperparameters as params, the trial's objective value as a metric, and the trial number as a tag. No boilerplate required inside your objective function.

snippet.py
# ─── optuna_mlflow_integration.py ─────────────────────────────────────────────
# Tune XGBoost with Optuna; automatically log every trial to MLflow.

import optuna
import mlflow
from optuna.integration.mlflow import MLflowCallback

mlflow.set_tracking_uri("file:./mlruns")
mlflow.set_experiment("xgb_optuna_mlflow")

# The callback creates one nested MLflow run per Optuna trial.
mlflow_callback = MLflowCallback(
    tracking_uri=mlflow.get_tracking_uri(),
    metric_name="cv_accuracy",
)

import numpy as np
from xgboost import XGBClassifier
from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import StratifiedKFold, cross_val_score

X, y = load_breast_cancer(return_X_y=True)
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)


@mlflow_callback.track_in_mlflow()
def objective(trial):
    params = {
        "learning_rate":    trial.suggest_float("learning_rate", 1e-3, 1.0, log=True),
        "max_depth":        trial.suggest_int("max_depth", 3, 10),
        "n_estimators":     trial.suggest_int("n_estimators", 50, 500),
        "subsample":        trial.suggest_float("subsample", 0.5, 1.0),
        "colsample_bytree": trial.suggest_float("colsample_bytree", 0.5, 1.0),
        "eval_metric":      "logloss",
        "tree_method":      "hist",
        "n_jobs":           -1,
        "random_state":     42,
    }
    model = XGBClassifier(**params)
    scores = cross_val_score(model, X, y, cv=cv, scoring="accuracy", n_jobs=-1)
    # Log additional metrics inside the objective — they land in the same run
    mlflow.log_metric("cv_accuracy_std", float(scores.std()))
    return float(scores.mean())


study = optuna.create_study(
    direction="maximize",
    sampler=optuna.samplers.TPESampler(seed=42),
    study_name="xgb_with_mlflow",
)
study.optimize(objective, n_trials=50, callbacks=[mlflow_callback])

print(f"Best trial: {study.best_trial.number}, value: {study.best_value:.4f}")
print(f"Best params: {study.best_params}")
# In MLflow UI, you can now filter by `tags.mlmParentRunId` to see all 50 trials.

With this integration, every Optuna trial becomes a first-class MLflow run that you can search, filter, and compare in the UI. If the production team asks "what hyperparameters did trial 342 use?", you open the MLflow UI, filter by trial number, and the answer is right there — params, metrics, tags, and any artifacts you logged.

4. The MLflow Model Registry: Versioning for Production

Logging a model as an artifact is step one. Step two is registering it in the Model Registry — a versioned catalog of models that supports staging ("None", "Staging", "Production", "Archived") and lets you promote a model from research to production with a single API call. This is what enables the workflow: "every Friday at 5pm, take the best model from this week's experiments and promote it to Staging for QA."

snippet.py
# ─── mlflow_registry.py ──────────────────────────────────────────────────────
# Register a trained model and transition it to the Production stage.

import mlflow
from mlflow.tracking import MlflowClient

mlflow.set_tracking_uri("file:./mlruns")

# After a tuning study, train the best model on all data and log it.
import xgboost as xgb
from sklearn.datasets import load_breast_cancer
import optuna

X, y = load_breast_cancer(return_X_y=True)
study = optuna.load_study(
    study_name="xgb_with_mlflow", storage="sqlite:///xgb.db"
)
best_params = study.best_params
best_params.update({"eval_metric": "logloss", "tree_method": "hist",
                    "n_jobs": -1, "random_state": 42})

with mlflow.start_run(run_name="champion_model_v1") as run:
    model = xgb.XGBClassifier(**best_params)
    model.fit(X, y)
    mlflow.log_params(best_params)
    mlflow.log_metric("val_accuracy", float(study.best_value))

    # Log the model WITH a registered name
    model_uri = mlflow.sklearn.log_model(
        model, artifact_path="model",
        registered_model_name="BreastCancerXGB",
    ).model_uri

    print(f"Run ID: {run.info.run_id}")
    print(f"Model URI: {model_uri}")

# Promote the latest version of the registered model to "Production"
client = MlflowClient()
versions = client.search_model_versions("name='BreastCancerXGB'")
latest = max(versions, key=lambda v: int(v.version))
client.transition_model_version_stage(
    name="BreastCancerXGB",
    version=latest.version,
    stage="Production",
    archive_existing_versions=True,  # auto-archive previously-Production versions
)
print(f"Promoted version {latest.version} to Production")

# Load the Production model from the registry (in your serving code)
import mlflow.sklearn
prod_model = mlflow.sklearn.load_model(
    model_uri="models:/BreastCancerXGB/Production"
)
print(f"Production model loaded. Test accuracy: {prod_model.score(X, y):.4f}")

The models:/BreastCancerXGB/Production URI is the magic ingredient: it always points to whatever version is currently in the "Production" stage. Your serving code never needs to know the version number — promote a new model, restart the server, and you are serving the new model. Roll back by transitioning the previous version back to Production.

5. Weights & Biases Sweeps: Real-Time Dashboards

W&B is a commercial SaaS (free tier is generous) that does everything MLflow does plus a few things MLflow doesn't: real-time dashboards that update as runs stream in, system-metric charts (GPU utilization, memory, disk) out of the box, sweep configs that look like YAML versions of Optuna search spaces, and an extremely polished UI. Most teams that try both end up on W&B for the dashboards alone.

The W&B equivalent of an Optuna study is a sweep: a YAML config that specifies the method (bayes, random, or grid), the metric to optimize, and the parameter ranges. You launch the sweep with wandb sweep config.yaml, then run agents with wandb agent <sweep_id> — agents can run on multiple machines in parallel, all reporting back to the same dashboard.

snippet.py
# ─── wandb_sweep_config.yaml ──────────────────────────────────────────────────
# Save this as sweep.yaml, then:
#   wandb sweep sweep.yaml             # prints sweep_id
#   wandb agent <sweep_id>             # runs trials (run on N machines for parallel)

program: train_xgb.py
method: bayes                    # Bayesian optimization (uses Hyperopt's TPE under the hood)
metric:
  name: cv_accuracy
  goal: maximize
parameters:
  learning_rate:
    distribution: log_uniform_values
    min: 0.001
    max: 1.0
  max_depth:
    distribution: int_uniform
    min: 3
    max: 12
  n_estimators:
    distribution: int_uniform
    min: 50
    max: 500
  subsample:
    distribution: uniform
    min: 0.5
    max: 1.0
  colsample_bytree:
    distribution: uniform
    min: 0.5
    max: 1.0
  min_child_weight:
    distribution: int_uniform
    min: 1
    max: 10
early_terminate:
  type: hyperband
  min_iter: 5
  eta: 2
  s: 0

The training script (train_xgb.py) is short — it just calls wandb.init(), reads the parameter values from wandb.config, trains the model, and calls wandb.log():

snippet.py
# ─── train_xgb.py ─────────────────────────────────────────────────────────────
# W&B sweep agent entry point. One invocation = one trial.

import wandb
import numpy as np
from xgboost import XGBClassifier
from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import StratifiedKFold, cross_val_score

def main():
    wandb.init(project="xgb-breast-cancer")
    config = wandb.config

    X, y = load_breast_cancer(return_X_y=True)
    cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

    model = XGBClassifier(
        learning_rate=config.learning_rate,
        max_depth=config.max_depth,
        n_estimators=config.n_estimators,
        subsample=config.subsample,
        colsample_bytree=config.colsample_bytree,
        min_child_weight=config.min_child_weight,
        eval_metric="logloss",
        tree_method="hist",
        n_jobs=-1,
        random_state=42,
    )
    scores = cross_val_score(model, X, y, cv=cv, scoring="accuracy", n_jobs=-1)

    wandb.log({
        "cv_accuracy": float(scores.mean()),
        "cv_accuracy_std": float(scores.std()),
    })

if __name__ == "__main__":
    main()

The combination of method: bayes (W&B's TPE implementation) and the hyperband early-termination rule makes W&B Sweeps a serious competitor to Optuna. The main trade-off vs. Optuna: W&B Sweeps is easier to run across multiple machines and has a much better UI, but Optuna has more samplers (CMA-ES, QNSCV, NSGA-II for multi-objective) and gives you more programmatic control.

💡
W&B Sweeps vs Optuna: when to pick which

Use W&B Sweeps if your team already uses W&B for experiment tracking, you want real-time dashboards, and your search needs are covered by TPE / random / grid. Use Optuna if you need multi-objective optimization (NSGA-II), pruners (Hyperband, SuccessiveHalving), programmatic search-space construction (conditional parameters, dynamic search spaces), or you cannot ship data to a SaaS. Many teams use both: Optuna for the heavy algorithmic lifting, W&B for the dashboarding.

6. Ray Tune: Distributed Tuning at Cluster Scale

When your tuning job no longer fits on one machine — either because each trial is too expensive (e.g., training a transformer for an hour) or because you want thousands of trials — you need a distributed tuner. Ray Tune is the leading option: it runs trials in parallel across a Ray cluster (which can span multiple machines and GPUs), integrates with Optuna's samplers and Hyperband's pruners, and is fault-tolerant (if a node dies, its trials resume from the last checkpoint on another node).

The Ray Tune API is built around tune.run() (or the newer Tuner API) and a set of search-space primitives: tune.choice, tune.uniform, tune.loguniform, tune.randint, tune.quniform. The same code runs on a single machine or a 100-node cluster — the only change is how you start the Ray cluster.

snippet.py
# ─── ray_tune_xgb.py ──────────────────────────────────────────────────────────
# Distributed XGBoost tuning with Ray Tune + Optuna sampler + ASHA pruner.

import os
import ray
from ray import tune
from ray.tune.search.optuna import OptunaSearch
from ray.tune.schedulers import ASHAScheduler
from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import StratifiedKFold, cross_val_score
from xgboost import XGBClassifier

# Initialize Ray (omit for local; pass address='ray://...' for a cluster)
ray.init(ignore_reinit_error=True)

# Load data once and put in Ray's object store so workers don't re-load it
data = load_breast_cancer(return_X_y=True)
X_ref = ray.put(data[0])
y_ref = ray.put(data[1])


def trainable(config):
    """One trial = one call to this function. Runs on a Ray worker."""
    X = ray.get(X_ref)
    y = ray.get(y_ref)
    cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

    model = XGBClassifier(
        learning_rate=config["learning_rate"],
        max_depth=config["max_depth"],
        n_estimators=config["n_estimators"],
        subsample=config["subsample"],
        colsample_bytree=config["colsample_bytree"],
        min_child_weight=config["min_child_weight"],
        eval_metric="logloss",
        tree_method="hist",
        n_jobs=os.cpu_count(),
        random_state=42,
    )
    scores = cross_val_score(model, X, y, cv=cv, scoring="accuracy")
    # Report the mean score back to Tune; the scheduler can use this to prune
    tune.report(cv_accuracy=float(scores.mean()))


# Search space — Ray Tune style
search_space = {
    "learning_rate":     tune.loguniform(1e-3, 1.0),
    "max_depth":         tune.randint(3, 13),
    "n_estimators":      tune.randint(50, 501),
    "subsample":         tune.uniform(0.5, 1.0),
    "colsample_bytree":  tune.uniform(0.5, 1.0),
    "min_child_weight":  tune.randint(1, 11),
}

# Optuna provides the sampler; ASHA provides early termination of bad trials
search_alg = OptunaSearch(metric="cv_accuracy", mode="max")
scheduler = ASHAScheduler(
    max_t=1,                  # for CV: each trial is a single "step"
    grace_period=1,
    reduction_factor=2,
)

tuner = tune.Tuner(
    trainable,
    param_space=search_space,
    tune_config=tune.TuneConfig(
        metric="cv_accuracy",
        mode="max",
        search_alg=search_alg,
        scheduler=scheduler,
        num_samples=100,           # number of trials
        max_concurrent_trials=8,   # parallel workers
    ),
    run_config=ray.train.RunConfig(
        name="xgb_ray_tune",
        storage_path="/tmp/ray_results",
        checkpoint_config=ray.train.CheckpointConfig(
            checkpoint_frequency=1,         # fault-tolerance
            checkpoint_at_end=True,
        ),
    ),
)
results = tuner.fit()

best = results.get_best_result(metric="cv_accuracy", mode="max")
print(f"Best trial config: {best.config}")
print(f"Best cv_accuracy:  {best.metrics['cv_accuracy']:.4f}")

The ASHAScheduler (Asynchronous Successive Halving) is Ray Tune's analogue of Optuna's HyperbandPruner: it terminates unpromising trials early, which is the only way to make 100 trials of an expensive model fit in a reasonable wall-clock budget. The OptunaSearch wrapper lets you use TPE (or any other Optuna sampler) inside Ray Tune — you get the algorithmic quality of Optuna with the distributed execution of Ray.

7. Reproducibility: The Hardest Problem in ML

"I trained the model last week and got 0.982 accuracy. I re-ran the same code today and got 0.979. What changed?" — every ML engineer, eventually. The answer is usually: nothing you wrote down. ML training is full of hidden non-determinism: thread scheduling in n_jobs=-1, GPU kernel selection, dataset shuffling order, cuDNN algorithm auto-tuning. Reproducibility is the discipline of pinning down as many of these as possible.

The minimum bar for reproducibility in a tuning job:

  • Set a global random seed at the top of every script: np.random.seed(42), random.seed(42), os.environ["PYTHONHASHSEED"] = "42". For PyTorch: torch.manual_seed(42), torch.cuda.manual_seed_all(42).
  • Pass random_state=42 to every scikit-learn estimator and CV splitter. This is non-negotiable — without it, StratifiedKFold(shuffle=True) will produce different folds every run.
  • Force deterministic cuDNN: torch.backends.cudnn.deterministic = True and torch.backends.cudnn.benchmark = False. This costs ~5–10% training speed but eliminates one major source of GPU non-determinism.
  • Pin your environment: capture pip freeze (or conda env export) at the start of every experiment and log it as an MLflow artifact. A version bump in numpy or xgboost can silently change numerical results.
  • Pin your data: log a hash of the dataset (e.g., hashlib.sha256(X.tobytes()).hexdigest()) so you can detect data drift between runs.
snippet.py
# ─── reproducibility_boilerplate.py ──────────────────────────────────────────
# Drop this at the top of every tuning script.

import os
import random
import hashlib

import numpy as np

SEED = 42

# 1. Python hash seed (affects dict ordering in some CPython versions)
os.environ["PYTHONHASHSEED"] = str(SEED)

# 2. Python's random module (used by some libraries under the hood)
random.seed(SEED)

# 3. NumPy (the big one — affects cross_val_score fold assignment, bootstrapping, etc.)
np.random.seed(SEED)

# 4. PyTorch (uncomment if you use it)
# import torch
# torch.manual_seed(SEED)
# torch.cuda.manual_seed_all(SEED)
# torch.backends.cudnn.deterministic = True
# torch.backends.cudnn.benchmark = False

# 5. Optional: force single-threaded NumPy for bit-exact reproducibility
# (costs speed; only do this when bit-exactness matters, e.g., publications)
# os.environ["OMP_NUM_THREADS"] = "1"
# os.environ["MKL_NUM_THREADS"] = "1"

# 6. Log a hash of the dataset so you can detect data drift later
from sklearn.datasets import load_breast_cancer
X, y = load_breast_cancer(return_X_y=True)
data_hash = hashlib.sha256(X.tobytes() + y.tobytes()).hexdigest()
print(f"Dataset SHA-256: {data_hash}")

# Then pass random_state=SEED everywhere you construct an estimator or splitter
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import StratifiedKFold

cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=SEED)
model = GradientBoostingClassifier(random_state=SEED)

Even with all of this, CUDA adds non-determinism that you cannot fully eliminate. Some GPU operations (atomic adds in reductions, non-deterministic reduction algorithms in cuDNN) produce different floating-point results on different runs even with deterministic flags set. For 99% of applications this is fine — the differences are at the 4th decimal place of accuracy — but if you need bit-exact reproducibility (e.g., for a regulated industry), you must run on CPU. There is no other way.

⚠️
Reproducibility is a moving target

Reproducibility flags differ across library versions. torch.use_deterministic_algorithms(True) (PyTorch 1.8+) is stricter than cudnn.deterministic alone but will error out on operations that have no deterministic implementation. Always check the reproducibility docs for the exact library versions you are using — and pin those versions in your requirements.txt.

8. Cost-Aware Optimization: Accuracy Is Not the Only Objective

In research, you optimize one metric: accuracy. In production, you almost always care about at least two: accuracy and inference latency. A model that is 0.5% more accurate but takes 50ms per inference instead of 1ms may be useless if your SLA is "respond in under 10ms." The production-grade answer is multi-objective optimization: search for the Pareto front of (accuracy, latency) and pick an operating point on it.

The Pareto front is the set of trials where no other trial is better on both objectives simultaneously. A trial is dominated if there exists another trial with higher accuracy and lower latency; the Pareto front is the set of undominated trials. Visually, it is the lower-left boundary of the scatter plot of (latency, accuracy) — see the figure below.

Conceptual Pareto front for (latency, accuracy). Green points are Pareto-optimal; gray points are dominated. Two natural operating points: A = best accuracy subject to latency < 1ms SLA; B = lowest latency subject to accuracy > 0.97. Either is a defensible production choice.

Optuna supports multi-objective optimization natively via the NSGAIISampler (Non-dominated Sorting Genetic Algorithm II). Your objective function returns a tuple of values instead of a single scalar, and you create the study with directions=["maximize", "minimize"]. Optuna maintains the Pareto front as the study progresses.

$$ \mathcal{{P}} = \bigl\{{ (\text{{acc}}_i, \text{{lat}}_i) \;\big|\; \neg\exists j: \text{{acc}}_j \geq \text{{acc}}_i \,\land\, \text{{lat}}_j \leq \text{{lat}}_i \bigr\}} $$
Eq. 1 — Pareto front $\mathcal{P}$: the set of trials not dominated by any other trial.
snippet.py
# ─── multi_objective_optuna.py ────────────────────────────────────────────────
# Multi-objective tuning: maximize accuracy AND minimize inference latency.

import time
import optuna
import numpy as np
from xgboost import XGBClassifier
from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import StratifiedKFold, cross_val_score

X, y = load_breast_cancer(return_X_y=True)
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)


def measure_latency(model, X, n_repeats=5, n_samples=1000):
    """Median per-sample inference latency in milliseconds."""
    rng = np.random.default_rng(42)
    idx = rng.choice(len(X), size=min(n_samples, len(X)), replace=False)
    Xs = X[idx]
    # Warm up
    for _ in range(10):
        model.predict(Xs)
    # Measure
    times = []
    for _ in range(n_repeats):
        t0 = time.perf_counter()
        for _ in range(100):
            model.predict(Xs)
        t1 = time.perf_counter()
        times.append((t1 - t0) / 100 / len(Xs) * 1000)  # ms per sample
    return float(np.median(times))


def objective(trial):
    params = {
        "learning_rate":     trial.suggest_float("learning_rate", 1e-3, 1.0, log=True),
        "max_depth":         trial.suggest_int("max_depth", 3, 10),
        "n_estimators":      trial.suggest_int("n_estimators", 50, 500),
        "subsample":         trial.suggest_float("subsample", 0.5, 1.0),
        "colsample_bytree":  trial.suggest_float("colsample_bytree", 0.5, 1.0),
        "min_child_weight":  trial.suggest_int("min_child_weight", 1, 10),
        "eval_metric":       "logloss",
        "tree_method":       "hist",
        "n_jobs":            1,           # single-threaded for fair latency measurement
        "random_state":      42,
    }
    model = XGBClassifier(**params)
    scores = cross_val_score(model, X, y, cv=cv, scoring="accuracy")
    acc = float(scores.mean())

    model.fit(X, y)
    lat = measure_latency(model, X)

    # Return BOTH objectives as a tuple
    return acc, lat


study = optuna.create_study(
    directions=["maximize", "minimize"],
    sampler=optuna.samplers.NSGAIISampler(seed=42),
    study_name="xgb_multi_obj",
    storage="sqlite:///xgb_multiobj.db",
)
study.optimize(objective, n_trials=100)

# Get the Pareto front
pareto = study.best_trials
print(f"Pareto front: {len(pareto)} trials")
for t in pareto[:10]:
    acc, lat = t.values
    print(f"  trial {t.number:3d}  acc={acc:.4f}  lat={lat:.4f} ms  params={t.params}")

# Pick operating points
A = max(pareto, key=lambda t: t.values[0] if t.values[1] < 1.0 else -1)
# More explicitly: among trials with lat < 1ms, find the one with max acc
feasible = [t for t in pareto if t.values[1] < 1.0]
if feasible:
    op_a = max(feasible, key=lambda t: t.values[0])
    print(f"\nOperating point A (max acc, lat<1ms): "
          f"acc={op_a.values[0]:.4f} lat={op_a.values[1]:.4f}")

feasible_b = [t for t in pareto if t.values[0] > 0.97]
if feasible_b:
    op_b = min(feasible_b, key=lambda t: t.values[1])
    print(f"Operating point B (min lat, acc>0.97): "
          f"acc={op_b.values[0]:.4f} lat={op_b.values[1]:.4f}")

The choice between operating point A and operating point B is not a technical decision — it is a product decision. A is the right choice when accuracy is paramount and the latency SLA is a hard constraint (e.g., a medical screening tool where you would rather be slow than wrong). B is the right choice when latency is the dominant cost (e.g., a real-time bidding model where a 1ms delay costs money). The Pareto front makes the tradeoff visible, which is the prerequisite for making it deliberately instead of by accident.

🧠 Quick Check

Your multi-objective Optuna study returns a Pareto front with 18 trials. The production team wants you to 'just give them the best model'. How do you respond?

Show Answer

Push back: there is no single 'best model' on a Pareto front — every point is, by definition, undominated. Ask the production team to specify a constraint (e.g., 'latency must be under 2ms') or a target (e.g., 'accuracy must be at least 0.97'). That constraint picks a unique operating point on the front. If they truly cannot specify one, hand them the entire front as a table and let them choose — but do not pick a single model on their behalf, because that is a product decision disguised as a technical one.

9. The Production Tuning Stack: Putting It All Together

A production-grade tuning system combines everything from this article: a sampler (Optuna or W&B Sweeps), a tracker (MLflow or W&B), a distributed executor (Ray Tune, if needed), reproducibility boilerplate, and — for most real applications — multi-objective optimization. The good news is that these tools compose cleanly: Optuna for the sampler, MLflow for the tracking, Ray Tune for the execution, all in one script.

snippet.py
# ─── production_stack.py ─────────────────────────────────────────────────────
# Optuna (sampler) + Ray Tune (executor) + MLflow (tracker) + reproducibility.

import os
import random
import hashlib
import numpy as np

SEED = 42
os.environ["PYTHONHASHSEED"] = str(SEED)
random.seed(SEED)
np.random.seed(SEED)

import ray
from ray import tune
from ray.tune.search.optuna import OptunaSearch
from ray.tune.schedulers import ASHAScheduler
from ray.tune.integration.mlflow import mlflow_mixin
import mlflow

import mlflow.sklearn
from xgboost import XGBClassifier
from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import StratifiedKFold, cross_val_score
from sklearn.metrics import accuracy_score

mlflow.set_tracking_uri("file:./mlruns")
mlflow.set_experiment("xgb_production_stack")

# Pin data
X, y = load_breast_cancer(return_X_y=True)
data_hash = hashlib.sha256(X.tobytes() + y.tobytes()).hexdigest()
print(f"Dataset hash: {data_hash}")

ray.init(ignore_reinit_error=True)
X_ref, y_ref = ray.put(X), ray.put(y)


@mlflow_mixin
def trainable(config):
    X = ray.get(X_ref)
    y = ray.get(y_ref)
    cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=SEED)

    model = XGBClassifier(
        learning_rate=config["learning_rate"],
        max_depth=config["max_depth"],
        n_estimators=config["n_estimators"],
        subsample=config["subsample"],
        colsample_bytree=config["colsample_bytree"],
        min_child_weight=config["min_child_weight"],
        eval_metric="logloss", tree_method="hist",
        n_jobs=1, random_state=SEED,
    )
    scores = cross_val_score(model, X, y, cv=cv, scoring="accuracy")
    acc = float(scores.mean())
    # MLflow logging happens automatically via the @mlflow_mixin decorator
    mlflow.log_metric("cv_accuracy", acc)
    mlflow.log_metric("cv_accuracy_std", float(scores.std()))
    tune.report(cv_accuracy=acc)


search_space = {
    "learning_rate":     tune.loguniform(1e-3, 1.0),
    "max_depth":         tune.randint(3, 13),
    "n_estimators":      tune.randint(50, 501),
    "subsample":         tune.uniform(0.5, 1.0),
    "colsample_bytree":  tune.uniform(0.5, 1.0),
    "min_child_weight":  tune.randint(1, 11),
    # mlflow_mixin requires this key
    "mlflow": {"experiment_name": "xgb_production_stack", "tracking_uri": "file:./mlruns"},
}

tuner = tune.Tuner(
    trainable,
    param_space=search_space,
    tune_config=tune.TuneConfig(
        metric="cv_accuracy", mode="max",
        search_alg=OptunaSearch(),
        scheduler=ASHAScheduler(max_t=1, grace_period=1),
        num_samples=200, max_concurrent_trials=8,
    ),
)
results = tuner.fit()
best = results.get_best_result(metric="cv_accuracy", mode="max")
print(f"Best cv_accuracy: {best.metrics['cv_accuracy']:.4f}")
print(f"Best config: {best.config}")

This is the production stack: 200 trials, distributed across 8 workers, with Optuna TPE doing the sampling, ASHA pruning bad trials, MLflow logging every trial, and reproducibility flags pinned at the top. On a single beefy machine it finishes in ~30 minutes; on a small cluster, in ~5 minutes. Every trial is in the MLflow UI, queryable by parameter or metric. If the production team asks "what did trial 167 do?", the answer is two clicks away.

🎯 Key Takeaways

  • Tracking is not optional at scale. Print statements work for 5 trials; MLflow or W&B is required for 50+.
  • MLflow is the open-source standard — start_run, log_params, log_metrics, log_artifact. The Model Registry adds versioning and staging.
  • W&B Sweeps is the commercial alternative — better dashboards, easier multi-machine, fewer sampler choices.
  • Ray Tune is the distributed executor when one machine is not enough. Pairs cleanly with Optuna samplers and ASHA pruning.
  • Reproducibility is hard — set seeds everywhere, pin your environment, hash your data, and accept that CUDA non-determinism is the floor.
  • Multi-objective optimization with NSGA-II gives you a Pareto front of (accuracy, latency). The product team picks the operating point; the engineer delivers the front.
  • The tools compose: Optuna + Ray Tune + MLflow in one script gives you algorithmic quality, distributed execution, and queryable tracking simultaneously.
