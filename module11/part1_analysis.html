1. Why Analysis Is Half the Work

Running a hyperparameter search is the easy part. Modern tools like Optuna, Ray Tune, and Hyperopt will happily spend hours of GPU time exploring a search space and hand you back a "best trial" at the end. But the best trial is just one row of a much richer dataset: every trial you ran is a sample of the function $f(\lambda) = \text{validation score}(\lambda)$, and that function contains everything you could want to know — which hyperparameters matter, where the sweet spots live, which combinations are quietly destroying your model, and whether your search has converged or is still finding new regions. Throwing those samples away and keeping only the argmax is like running a clinical trial and reporting only the mean.

This article is the analytical companion to Module 10 (where we covered the search algorithms themselves). We assume you already have an Optuna study object in memory, or a scikit-learn cv_results_ dictionary from a GridSearchCV or RandomizedSearchCV run. From those two starting points we will build a complete analysis pipeline: convergence plots, parallel coordinates, hyperparameter importance via fANOVA, contour and slice plots, distributional analysis of cv_results_, and a train-vs-validation overfitting check. By the end you should be able to look at a finished study and tell, in five minutes, what worked, what didn't, and what to try next.

The unifying principle behind every plot in this article is the same: a tuning study is a dataset, and the same exploratory-data-analysis toolkit you would apply to any dataset applies here too. The "rows" are trials, the "columns" are hyperparameters plus the score, and the relationships between columns are exactly what we want to visualize. The only wrinkle is that the columns are heterogeneous — some are continuous, some are categorical, some are log-scale — and the visualizations need to respect that.

2. Convergence Plots: When to Stop Spending Compute

The single most practically important plot is the convergence plot: best-so-far validation score versus trial number. It answers the question every ML engineer asks during a long search: is this still worth running? The shape of the curve tells you.

If the curve is still climbing at the right edge of the plot — meaning the best score has improved in the last 5–10 trials — your search is still finding useful new regions and you should let it run longer. If the curve has flattened into a long horizontal asymptote — meaning the best score has not improved in the last 30+ trials — your search has converged and additional trials are unlikely to beat the incumbent. Bayesian optimizers like TPE and CMA-ES converge in a characteristic way: rapid improvement in the first 10–20 trials as the surrogate model is built, then a long tail of small improvements. Random search converges more slowly and more linearly. Grid search doesn't really "converge" — it just stops when the grid is exhausted.

Generating this plot from an Optuna study is a one-liner via the built-in visualization module:

snippet.py
# ─── convergence_plot.py ──────────────────────────────────────────────────────
# Generate the best-score-vs-trial-number curve from an Optuna study.
# This is the single most important diagnostic for "should I keep running?"

import optuna
import matplotlib.pyplot as plt

study = optuna.load_study(study_name="xgb_breast_cancer", storage="sqlite:///xgb.db")

# Built-in plotly version (interactive):
fig = optuna.visualization.plot_optimization_history(study)
fig.write_html("optimization_history.html")
fig.write_image("optimization_history.png", width=900, height=500)

# Manual matplotlib version (for static reports):
trials = [t for t in study.trials if t.state == optuna.trial.TrialState.COMPLETE]
values = [t.value for t in trials]                       # per-trial score
best_so_far = []
running_best = float("-inf")
for v in values:
    running_best = max(running_best, v)
    best_so_far.append(running_best)

fig, ax = plt.subplots(figsize=(9, 4.5), constrained_layout=True)
ax.plot(range(1, len(values) + 1), values, "o", color="#8899aa",
        alpha=0.4, markersize=4, label="per-trial score")
ax.plot(range(1, len(values) + 1), best_so_far, "-", color="#38bdf8",
        linewidth=2.5, label="best so far")
ax.set_xlabel("Trial number")
ax.set_ylabel("Validation accuracy")
ax.set_title("Optimization history")
ax.legend(loc="lower right")
plt.savefig("convergence_manual.png", dpi=120)
plt.show()

# Quantitative convergence check: how many trials since last improvement?
import numpy as np
idx_best = int(np.argmax(values))
trials_since_improvement = len(values) - 1 - idx_best
print(f"Best score = {max(values):.4f} at trial {idx_best + 1}")
print(f"Trials since last improvement: {trials_since_improvement}")
print(f"Converged? {'yes' if trials_since_improvement > 30 else 'maybe keep running'}")

Note the quantitative convergence check at the bottom. Eyeballing the curve is fine for a quick look, but if you are automating search budgets (e.g. "stop when converged"), you want a number. A common rule of thumb: if the best score has not improved in the last max(30, 10% × n_trials) trials, declare the search converged. This is also what Optuna's TrialPruner uses internally for early stopping of unpromising trials.

⚠️
Convergence != optimality

A flat convergence curve means the search algorithm has stopped finding improvements — it does not mean you have found the global optimum. The true optimum might live in a region the sampler never explored. If your convergence plot is flat but the score is still unsatisfying, try a different sampler (switch TPE → CMA-ES, or restart TPE with a fresh seed) or expand the search space before declaring defeat.

3. Parallel Coordinates: The Bird's-Eye View

The parallel coordinates plot is the most information-dense visualization in the hyperparameter analyst's toolkit. Each vertical axis represents one hyperparameter (with its values min-max normalized onto [0, 1] for display), and each line connecting the axes represents one trial, colored by its score. The result is a tangle of lines that, when you squint, reveals where in the joint space the good trials cluster.

Conceptual parallel coordinates plot. Each line is one trial; line color encodes the validation score (green=best, pink=worst). The green trials cluster in a tight region across all five axes — that is the sweet spot.

The plot above is a schematic with hand-drawn lines, but the real thing — generated by optuna.visualization.plot_parallel_coordinate(study) — looks essentially the same. The diagnostic value is in the patterns: if the green (best) lines all pass through a narrow band on the learning_rate axis, that band is your sweet spot for learning_rate. If the green lines are scattered across the entire subsample axis, subsample barely matters and you can stop tuning it.

The single most common finding from a parallel coordinates plot is an interaction: two hyperparameters that look unimportant in isolation but jointly determine success. The classic example is learning_rate and n_estimators in gradient boosting — either alone looks like noise, but together they form a tight diagonal band of good trials (low LR + many trees, or higher LR + fewer trees). Parallel coordinates makes this band jump off the page.

snippet.py
# ─── parallel_coords.py ───────────────────────────────────────────────────────
# Generate parallel coordinates plot from an Optuna study.

import optuna

study = optuna.load_study(study_name="xgb_breast_cancer", storage="sqlite:///xgb.db")

# Plotly version (interactive, hover to see exact values):
fig = optuna.visualization.plot_parallel_coordinate(
    study,
    params=["learning_rate", "max_depth", "n_estimators",
            "subsample", "colsample_bytree", "min_child_weight"],
)
fig.write_html("parallel_coords.html")
fig.show()

# To restrict to the top-10% trials only (cleaner visualization):
import numpy as np
trials = [t for t in study.trials if t.state == optuna.trial.TrialState.COMPLETE]
values = [t.value for t in trials]
threshold = np.quantile(values, 0.90)
top_trials = [t for t in trials if t.value >= threshold]
print(f"Plotting {len(top_trials)} top trials out of {len(trials)} total")

# Optuna 3.x lets you filter by passing a sub-study:
top_study = optuna.create_study()
for t in top_trials:
    top_study.add_trial(t)
fig_top = optuna.visualization.plot_parallel_coordinate(top_study)
fig_top.write_html("parallel_coords_top10pct.html")

The two-pass trick at the bottom — generate the plot once with all trials, then again with only the top 10% — is the most useful practical tip in this article. The full plot is overwhelming on studies with more than 100 trials; the top-10% plot shows you exactly which slice of each axis the best trials live in, with no noise.

4. Hyperparameter Importance: fANOVA and What Actually Matters

Not every hyperparameter is worth tuning. For most model families, two or three hyperparameters explain 80–95% of the variance in validation score, and the rest are noise. Knowing which is which lets you focus your next search budget on what actually matters, and lets you fix the noise hyperparameters to reasonable defaults.

The standard tool for quantifying this is fANOVA — functional ANOVA — originally introduced by Hutter et al. (2014) for analyzing AutoML competitions. The idea: treat the validation score as a function of all hyperparameters, decompose its variance into a sum of per-hyperparameter (and per-interaction) contributions, and report the fraction of variance each one explains. The decomposition is exact for additive functions and approximate (via a random-forest surrogate) for real validation surfaces. The output is a single bar chart: one bar per hyperparameter, height = fraction of variance explained.

$$ \text{{Var}}[f(\lambda)] = \sum_i V_i + \sum_{{i < j}} V_{{ij}} + \cdots \quad \text{{where}} \quad V_i = \text{{Var}}_{{\lambda_i}}\bigl[\mathbb{{E}}_{{\lambda_{{-i}}}}[f]\bigr] $$
Eq. 1 — fANOVA decomposition. $V_i$ is the marginal variance explained by $\lambda_i$ alone; $V_{ij}$ is the variance explained by the $\lambda_i \times \lambda_j$ interaction.

Optuna wraps this in a single function call:

snippet.py
# ─── param_importances.py ─────────────────────────────────────────────────────
# Compute and plot hyperparameter importances via fANOVA.

import optuna
from optuna.importance import get_param_importances, FanovaImportanceEvaluator

study = optuna.load_study(study_name="xgb_breast_cancer", storage="sqlite:///xgb.db")

# Quick & dirty: built-in plot
fig = optuna.visualization.plot_param_importances(study)
fig.write_html("param_importances.html")
fig.show()

# Programmatic access (returns an ordered dict):
importances = get_param_importances(study, evaluator=FanovaImportanceEvaluator(seed=42))
print("Hyperparameter importance (fraction of variance explained):")
for name, imp in importances.items():
    print(f"  {name:25s} {imp:.3f}")

# Typical output on XGBoost + breast_cancer with 100 trials:
#   learning_rate            0.512
#   max_depth                0.238
#   n_estimators             0.108
#   subsample                0.062
#   colsample_bytree         0.041
#   min_child_weight         0.018
#   reg_lambda               0.013
#   reg_alpha                0.008

The typical output above is a real pattern, not a contrived example. For XGBoost on a tabular classification task, learning_rate usually explains 40–60% of the variance, max_depth another 20–30%, and the remaining six or seven hyperparameters split the leftover 15%. This is exactly why modules 5 and 6 spent so much time on those two: they are not just "important," they are where the action is.

A subtle but critical caveat: importances are conditional on the search space you ran. If you set learning_rate to a fixed value, fANOVA will report it as unimportant — not because it doesn't matter, but because there's no variance in it to explain. Always read importances alongside the search space configuration.

💡
Compute importances on the second half of trials only

Bayesian samplers concentrate their queries in the good region after the first ~30% of trials. If you compute importances on the full study, the "good region" looks artificially over-represented and the importance estimates are biased. For more honest estimates, restrict to the last 50% of trials: get_param_importances(study, n_recent_trials=len(study.trials)//2).

5. Contour Plots: Spotting Pairwise Interactions

fANOVA tells you which hyperparameters matter individually; the contour plot tells you which pairs of hyperparameters interact. A contour plot is a 2D heatmap of the validation score as a function of two hyperparameters, with all other hyperparameters marginalized out. The shape of the contours reveals the interaction structure.

Round, concentric contours mean the two hyperparameters are independent: you can tune each one separately, in any order. Elongated diagonal contours mean they interact: the best value of one depends on the value of the other. A "ridge" pattern — a narrow good region running diagonally across the plot — means the interaction is strong and you cannot tune either hyperparameter without simultaneously considering the other.

The canonical interaction in gradient boosting is learning_rate × n_estimators: low LR with many trees is good, high LR with few trees is also good, but high LR with many trees overfits dramatically. A contour plot of those two parameters shows a diagonal ridge of good scores, with a hot red region in the "high LR + many trees" corner where every trial overfit. Seeing that ridge tells you to co-tune them — for example, fix n_estimators = 500 and tune learning_rate on a log scale.

snippet.py
# ─── contour_plot.py ──────────────────────────────────────────────────────────
# Generate contour plots for the top hyperparameter pairs.

import optuna
import itertools

study = optuna.load_study(study_name="xgb_breast_cancer", storage="sqlite:///xgb.db")

# Built-in: single pair
fig = optuna.visualization.plot_contour(
    study,
    params=["learning_rate", "max_depth"],
)
fig.write_html("contour_lr_depth.html")

# Built-in: full pairwise grid (every pair gets its own subplot)
fig_all = optuna.visualization.plot_contour(study)
fig_all.write_html("contour_all_pairs.html")

# Pick the top 3 most important params, then plot just those 3 pairs:
from optuna.importance import get_param_importances
imps = list(get_param_importances(study).keys())[:3]
for a, b in itertools.combinations(imps, 2):
    fig = optuna.visualization.plot_contour(study, params=[a, b])
    fig.write_html(f"contour_{a}_{b}.html")

# Manual matplotlib version (for a static report):
import numpy as np
import matplotlib.pyplot as plt
from scipy.interpolate import griddata

trials = [t for t in study.trials if t.state == optuna.trial.TrialState.COMPLETE]
lr = np.array([t.params["learning_rate"] for t in trials])
md = np.array([t.params["max_depth"] for t in trials], dtype=float)
val = np.array([t.value for t in trials])

grid_x, grid_y = np.meshgrid(
    np.linspace(lr.min(), lr.max(), 60),
    np.linspace(md.min(), md.max(), 60),
)
grid_z = griddata(np.column_stack([lr, md]), val, (grid_x, grid_y), method="cubic")

fig, ax = plt.subplots(figsize=(7, 5), constrained_layout=True)
cs = ax.contourf(grid_x, grid_y, grid_z, levels=20, cmap="viridis")
ax.scatter(lr, md, c=val, edgecolors="white", linewidth=0.5, s=30, cmap="viridis")
fig.colorbar(cs, label="Validation accuracy")
ax.set_xlabel("learning_rate")
ax.set_ylabel("max_depth")
ax.set_title("Contour: learning_rate × max_depth")
plt.savefig("contour_lr_depth_manual.png", dpi=120)
plt.show()

The matplotlib version uses scipy.interpolate.griddata to fit a smooth surface through the trial points. This is a quick-and-dirty surrogate; for a more principled surface use a Gaussian-process fit (Optuna does this internally for its GP-based samplers). Either way, the goal is the same: see the shape of the surface, not just the discrete trial points.

6. Slice Plots: 1D Sweet Spots, Marginalized

The slice plot is the natural reduction of a contour plot to one dimension: it shows validation score versus a single hyperparameter, with all other hyperparameters marginalized out (i.e., every trial plotted as a point, with no constraint on the other axes). It is the fastest way to answer "what's the right value for X?" for each hyperparameter in turn.

A good slice plot looks like a U-shape or a clear monotonic trend. A flat slice plot means the hyperparameter doesn't matter much (and fANOVA will agree). A slice plot with a clear valley — say, scores peaking at max_depth=6–9 and dropping on either side — tells you the sweet spot directly.

snippet.py
# ─── slice_plot.py ─────────────────────────────────────────────────────────────
# 1D marginal plots: validation score vs each hyperparameter individually.

import optuna

study = optuna.load_study(study_name="xgb_breast_cancer", storage="sqlite:///xgb.db")

# Built-in: all params at once (one subplot per hyperparameter)
fig = optuna.visualization.plot_slice(study)
fig.write_html("slice_all.html")

# Built-in: restrict to specific params and color by another param
fig = optuna.visualization.plot_slice(
    study,
    params=["learning_rate", "max_depth", "n_estimators"],
)
fig.write_html("slice_top3.html")

Slice plots complement contour plots: contour plots show interactions, slice plots show marginal effects. If a slice plot is flat but the corresponding hyperparameter shows up as important in fANOVA, that is a tell-tale sign of a strong interaction — the hyperparameter matters only in combination with something else, and you need the contour plot to find it.

7. Deep Analysis of cv_results_

If you ran a GridSearchCV or RandomizedSearchCV instead of Optuna, your starting point is the cv_results_ dictionary on the fitted search object. It contains, for every parameter combination evaluated, the per-fold test scores, the mean and std of those scores, the fit time, and the parameter values themselves. It is essentially a tidy DataFrame of your entire search.

Two analyses are particularly useful on cv_results_:

  • Box plots of score per categorical hyperparameter value. For each level of a categorical hyperparameter (e.g., kernel='rbf' vs kernel='poly'), collect all the per-fold scores across all other hyperparameter settings, and draw a box plot. If one level's box is consistently above the others, that level is robustly better — independent of the other knobs.
  • Score heatmaps for grid searches. If you ran a 2D grid over two hyperparameters, you can pivot cv_results_ into a 2D matrix and display it as a heatmap. This is exactly the contour plot, but for discrete grid points.
snippet.py
# ─── cv_results_analysis.py ───────────────────────────────────────────────────
# Analyze GridSearchCV / RandomizedSearchCV cv_results_ with pandas + seaborn.

import pandas as pd
import seaborn as sns
import matplotlib.pyplot as plt
from sklearn.datasets import load_breast_cancer
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import GridSearchCV, StratifiedKFold

X, y = load_breast_cancer(return_X_y=True)

# A small 2D grid: learning_rate × max_depth
grid = {
    "learning_rate": [0.01, 0.05, 0.1, 0.3, 1.0],
    "max_depth":     [2, 3, 5, 7, 9],
}
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
search = GridSearchCV(
    GradientBoostingClassifier(n_estimators=200, random_state=42),
    grid, cv=cv, scoring="accuracy", n_jobs=-1, return_train_score=True,
)
search.fit(X, y)

# Convert cv_results_ to a tidy DataFrame
df = pd.DataFrame(search.cv_results_)
print(df.columns.tolist()[:10])   # peek at available columns

# ─── 1. Box plot of CV accuracy by learning_rate level ──────────────────────
plt.figure(figsize=(9, 5))
sns.boxplot(data=df, x="param_learning_rate", y="mean_test_score",
            palette="viridis")
plt.xlabel("learning_rate")
plt.ylabel("Mean CV accuracy")
plt.title("CV accuracy distribution per learning_rate level")
plt.tight_layout()
plt.savefig("cv_box_lr.png", dpi=120)
plt.show()

# ─── 2. 2D heatmap of mean_test_score over (learning_rate, max_depth) ───────
pivot = df.pivot_table(
    index="param_learning_rate",
    columns="param_max_depth",
    values="mean_test_score",
)
plt.figure(figsize=(7, 5))
sns.heatmap(pivot, annot=True, fmt=".3f", cmap="viridis", square=True)
plt.title("Grid search heatmap: mean CV accuracy")
plt.tight_layout()
plt.savefig("cv_heatmap.png", dpi=120)
plt.show()

# ─── 3. Statistical check: does any level beat the others consistently? ─────
print("\nPer-level mean CV accuracy (sorted):")
print(df.groupby("param_learning_rate")["mean_test_score"]
        .agg(["mean", "std", "count"])
        .sort_values("mean", ascending=False))

The heatmap is essentially a discrete contour plot — it shows the same interaction structure, just without interpolation. The box plot is more useful when one of your hyperparameters is genuinely categorical (kernel choice, loss function), because the contour/slice machinery in Optuna assumes continuous parameters.

8. The Train-vs-Validation Overfitting Check

Every visualization so far has used validation score as the y-axis. But validation score alone can mislead you in one specific way: you can overfit to the validation set itself. If you run enough trials, eventually one of them will, by chance, have a high validation score that does not generalize to a truly held-out test set. The signal is in the train-vs-validation gap.

For each trial, plot train score (accuracy on the training folds) and validation score (accuracy on the held-out fold) against each other, colored by trial number or by validation score. The diagnostic pattern: if the best-validation trials have a train score that is much higher than their validation score — say, train = 1.000 and val = 0.975 — and the gap grows as you move to higher-validation-score trials, you are overfitting to the validation set. The cure is either more data (so the validation estimate is less noisy) or nested cross-validation (so the validation set you are tuning on is itself held out from a final evaluation).

snippet.py
# ─── train_vs_val.py ───────────────────────────────────────────────────────────
# Check whether the search is overfitting to the validation set.

import numpy as np
import matplotlib.pyplot as plt
from sklearn.datasets import load_breast_cancer
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import GridSearchCV, StratifiedKFold

X, y = load_breast_cancer(return_X_y=True)
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

# Note: return_train_score=True is required to get train scores per trial.
search = GridSearchCV(
    GradientBoostingClassifier(n_estimators=200, random_state=42),
    {
        "learning_rate": [0.01, 0.05, 0.1, 0.2, 0.5, 1.0],
        "max_depth":     [2, 3, 5, 7, 9, 12],
    },
    cv=cv, scoring="accuracy", n_jobs=-1, return_train_score=True,
)
search.fit(X, y)

import pandas as pd
df = pd.DataFrame(search.cv_results_)

# Scatter: train score (x) vs val score (y), colored by trial index
plt.figure(figsize=(7.5, 6))
sc = plt.scatter(
    df["mean_train_score"], df["mean_test_score"],
    c=df["mean_test_score"], cmap="viridis", s=50, edgecolors="white",
)
plt.plot([0.9, 1.0], [0.9, 1.0], "--", color="#8899aa", label="train = val (no overfit)")
plt.colorbar(sc, label="Validation accuracy")
plt.xlabel("Mean train accuracy (5-fold)")
plt.ylabel("Mean validation accuracy (5-fold)")
plt.title("Train vs validation: are top-val trials overfit?")
plt.legend(loc="lower right")
plt.tight_layout()
plt.savefig("train_vs_val.png", dpi=120)
plt.show()

# Quantitative check: gap = train - val, sorted by val score descending
df["gap"] = df["mean_train_score"] - df["mean_test_score"]
top10 = df.nlargest(10, "mean_test_score")[
    ["mean_train_score", "mean_test_score", "gap",
     "param_learning_rate", "param_max_depth"]
]
print("\nTop 10 trials by validation score:")
print(top10.to_string(index=False))

mean_gap = top10["gap"].mean()
overall_gap = df["gap"].mean()
print(f"\nMean train-val gap (top 10):   {mean_gap:.4f}")
print(f"Mean train-val gap (all trials): {overall_gap:.4f}")
if mean_gap > 1.5 * overall_gap:
    print("⚠️  Top-val trials show an overfitting signal. Consider nested CV.")
else:
    print("✓  No overfitting signal in top-val trials.")

The quantitative check at the end — comparing the mean train-val gap of the top-10 trials against the overall mean gap — is a useful heuristic. If the top-10 gap is more than 1.5× the overall gap, the search is selectively finding trials that overfit the validation folds. In that case, treat the reported best score as optimistic by the size of the gap, and validate the final model on a truly held-out test set (or via nested CV).

🔥
Nested CV is the gold standard

If you are reporting results in a paper or to a regulator, single-loop CV with hyperparameter tuning leaks — the validation set used to pick hyperparameters is the same one used to estimate generalization. Nested CV (5 outer folds × 3 inner folds = 15 fits, but it scales) fixes this by tuning separately inside each outer fold. Use it whenever the reported number matters more than the wall-clock cost.

9. The Full Analysis Pipeline: From Study to Report

Let's close with a single end-to-end script that takes an Optuna study and produces a complete analysis report: convergence, parallel coordinates, parameter importances, contour plots for the top-2 most important pairs, slice plots, and a train-vs-validation overfitting check. This is the script you should run after every tuning job before deciding what to do next.

snippet.py
# ─── analyze_study.py ─────────────────────────────────────────────────────────
# End-to-end analysis pipeline for an Optuna study.
# Produces: 6 HTML plots + a Markdown summary table.

import optuna
import itertools
import numpy as np
import pandas as pd
from pathlib import Path
from optuna.importance import get_param_importances, FanovaImportanceEvaluator

OUT = Path("analysis_report"); OUT.mkdir(exist_ok=True)
study = optuna.load_study(study_name="xgb_breast_cancer", storage="sqlite:///xgb.db")
trials = [t for t in study.trials if t.state == optuna.trial.TrialState.COMPLETE]
print(f"Loaded {len(trials)} complete trials. Best value = {study.best_value:.4f}")

# 1. Convergence (optimization history)
fig = optuna.visualization.plot_optimization_history(study)
fig.write_html(OUT / "01_optimization_history.html")

# 2. Parallel coordinates (all params)
fig = optuna.visualization.plot_parallel_coordinate(study)
fig.write_html(OUT / "02_parallel_coords_all.html")

# 3. Parallel coordinates (top 20% trials only)
threshold = np.quantile([t.value for t in trials], 0.80)
top_study = optuna.create_study(direction=study.direction)
for t in trials:
    if t.value >= threshold:
        top_study.add_trial(t)
fig = optuna.visualization.plot_parallel_coordinate(top_study)
fig.write_html(OUT / "03_parallel_coords_top20.html")

# 4. Hyperparameter importances
fig = optuna.visualization.plot_param_importances(study)
fig.write_html(OUT / "04_param_importances.html")
imps = get_param_importances(study, evaluator=FanovaImportanceEvaluator(seed=42))
top_params = list(imps.keys())[:3]
print(f"Top-3 most important params: {top_params}")

# 5. Contour plots for every pair of top-3 params (3 plots)
for a, b in itertools.combinations(top_params, 2):
    fig = optuna.visualization.plot_contour(study, params=[a, b])
    fig.write_html(OUT / f"05_contour_{a}_{b}.html")

# 6. Slice plots
fig = optuna.visualization.plot_slice(study, params=top_params)
fig.write_html(OUT / "06_slice_top3.html")

# 7. Train-vs-val overfitting check (requires user_attrs logged during the trial)
rows = []
for t in trials:
    row = {"number": t.number, "val_score": t.value}
    row.update(t.params)
    # 'train_score' must be stored via trial.set_user_attr("train_score", ...) in the objective
    row["train_score"] = t.user_attrs.get("train_score", np.nan)
    rows.append(row)
df = pd.DataFrame(rows)
if "train_score" in df and df["train_score"].notna().any():
    df["gap"] = df["train_score"] - df["val_score"]
    top10_gap = df.nlargest(10, "val_score")["gap"].mean()
    overall_gap = df["gap"].mean()
    overfit_flag = top10_gap > 1.5 * overall_gap
    print(f"Top-10 gap={top10_gap:.4f}  overall gap={overall_gap:.4f}  "
          f"overfit={'YES' if overfit_flag else 'no'}")

# 8. Markdown summary
with open(OUT / "summary.md", "w") as f:
    f.write(f"# Analysis Report\n\n")
    f.write(f"- Trials: {len(trials)}\n")
    f.write(f"- Best value: {study.best_value:.4f}\n")
    f.write(f"- Best params: {study.best_params}\n\n")
    f.write(f"## Hyperparameter importances\n\n")
    for name, imp in imps.items():
        bar = "#" * int(imp * 50)
        f.write(f"- `{name}`: {imp:.3f} {bar}\n")
    f.write(f"\n## Plots\n\n")
    for p in sorted(OUT.glob("*.html")):
        f.write(f"- [{p.name}]({p.name})\n")
print(f"\nReport written to {OUT.resolve()}")

Save this as analyze_study.py next to your tuning script and run it whenever a search finishes. The Markdown summary is small enough to commit to git alongside the study database, giving you a reproducible record of what each search found and why.

🧠 Quick Check

Your fANOVA importances show that `reg_lambda` explains only 0.5% of the variance in validation score. Should you (a) delete it from the search space, (b) keep it but fix it to a sensible default, or (c) leave it as-is in the next search?

Show Answer

Probably (b). A hyperparameter with <1% importance in this search is unlikely to start mattering in a larger search, so spending sampler budget on it is wasteful. But deleting it entirely is risky: fANOVA importances are conditional on the search space you ran, and `reg_lambda` might matter more on a different dataset. Fixing it to a reasonable default (e.g. reg_lambda=1.0, the XGBoost default) keeps it in the model without consuming tuning budget. Option (c) is acceptable if you have compute to spare; the sampler will simply learn to ignore it.

🎯 Key Takeaways

  • A tuning study is a dataset. Every trial is a row; treat the study with the same EDA toolkit you would apply to any other dataset.
  • Convergence plots tell you when to stop — flat best-so-far curve means the sampler has converged; still-rising curve means run more trials.
  • Parallel coordinates show the joint structure of the good region. Plot the top-10% trials separately for a cleaner view.
  • fANOVA gives you the importance ranking in one number per hyperparameter. Usually 2–3 hyperparameters explain 80%+ of variance.
  • Contour plots reveal pairwise interactions; slice plots show marginal 1D sweet spots. Use both — they are complementary.
  • Always check the train-vs-val gap for your top trials. A growing gap is the tell-tale sign of overfitting to the validation set.
  • Automate the analysis — run a single end-to-end script after every search and commit the report alongside the study database.
